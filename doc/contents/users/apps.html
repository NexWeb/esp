Creating ESP Applications

To start a new web application, first create a new directory named for the application then run esp init.

$ mkdir demo $ cd demo $ esp init

This will prepare your application to run ESP by generating two files and creating a dist directory:

  • The package.json file to describe the name and version of your application with any installed packages.
  • The esp.json file to configure ESP and HTTP.
  • The dist directory is the distribution directory for web content.

You can now create web pages to be served by ESP. Create a test ESP page under the dist directory called dist/index.esp with the contents:

<html> <body> <h1>Hello World</h1> </body> </html>

If a directory named dist is not present, esp will serve web content from the current directory.

Running

To run your application, start esp to serve browser requests.

$ esp serve

You can also just run a plain esp command without arguments.

$ esp

You can now browse to http://127.0.0.1:4000/index.esp to see the new page. The first time, there will be a brief pause as the page is compiled and saved. Next time, the page will be served from memory. If you modify the page, it will be transparently re-compiled.

Skeletons

There is another way to create ESP applications and that is by installing ESP skeletons. Skeletons are integrated starting points for ESP applications that include starter web pages, layouts, partial pages, stylesheets, scripts and required dependent packages for an easy, one-step launch of your web application. For example:

$ pak install esp-html-skeleton

This installs a skeleton application for an ESP HTML MVC application. Read more about skeletons in Application Skeletons.

Configuration Files

The two generated configuration files package.json and esp.json play a central role in configuring and evolving your application. Read more about these files in Configuring ESP.

Directories

The esp init and pak commands will create directories as required when creating an application, installing packages and generating components.

Directory Purpose
cache Cached compiled controllers and pages
contents Input web content source directory used by Expansive when rendering into the dist directory.
dist Directory of web content to serve to browser clients. Files outside the dist directory are not visible to the client. If a dist directory does not exist, files in the current directory will be served.
dist/assets Client side assets: images and media
dist/css Client side CSS
controllers Controller source code
db Databases and scripts
layouts Layout template pages used by Expansive.
paks Installed packs
partials Partial ESP pages used by Expansive
src Other application C source code
